## Surprise findings (not anticipated in CONTEXT pre-class)

1. **Row #37 UI contract vaporware** (A2#1) — the Phase 08 shape-consolidation dropped `ActiveSessionState.ErrorState` with a docstring promise that UI consumers would pattern-match `asyncValue.error` runtime types to route `GpsError` → recovery screens. Grep confirmed NO such UI code exists: `SessionDetailScreen._handleStart` stringifies errors into a generic red Text widget, and banner/burger/map widgets hide on non-Tracking state. This is a classic shape-consolidation regression — the controller is cleaner, the UI promises nothing has changed, and the user-visible recovery UX is the casualty. Surfaced by Agent #2 and triaged as the single Blocker of Phase 08.1.

2. **Pause-then-Cancel state stranding** (A1#1) — a second-order consequence of row #1's break-on-pause fix. Once the processing loop exits on `_pauseRequested`, `_processingDone` has fired; a subsequent `cancelActive()` awaits an already-completed Future and returns silently without emitting `DownloadCancelled` or cleaning staging. No test covers this transition.

3. **MapCamera echo-window second-echo mis-classification** (A2#2) — MapLibre's `onCameraIdle` can emit >1 event per programmatic move under style-reload paths (observed in 2026-04-22 device smoke). Row #35's "clear `_lastProgrammaticMoveAt` on first echo" design works in the single-echo case but drops follow-me under multi-echo — fix-on-fix risk on the narrow "setStyle → move → setStyle" rapid sequence path.
