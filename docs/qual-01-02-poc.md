# Phase 05 — QUAL-01 / QUAL-02 POC evidence log

Real-device background-tracking evidence for Phase 05 GPS & Session
Lifecycle. This file is the single source of truth for the POC walks
required by the Phase 05 Success Criteria #1 (Android OEM 30-min
background walk) and #2 (iOS 30-min background walk).

Each entry below documents one real-device walk: device + build + measured
stats + verdict. The PNG plots generated by `tool/plot_session_fixes.py`
live in `docs/poc-artifacts/` and are referenced by path from each entry.

## Protocol

1. Launch MirkFall on the target device.
2. Tap the FAB `+` and create a new session (e.g. "POC 2026-04-DD
   Pixel 4a"). Accept the rationale-screen pre-prompt.
3. Tap Start. Step through the two-step permission flow (While Using
   -> Always). On Android OEM builds with aggressive battery savers,
   step through the OEM guidance screen.
4. Confirm the persistent notification appears in the system tray
   ("MirkFall - <session name>").
5. Lock the screen. Put the phone in a pocket.
6. Walk for at least 30 minutes. A mix of walk + short transit + brief
   pause is fine and representative of real-world use.
7. Return, unlock, re-open the app. Confirm the session is still on the
   SessionDetailScreen (or re-enter via the notification tap).
8. Tap Stop. Confirm the notification disappears immediately.
9. Pull the database off the device:
   - **Android:** `adb pull /data/user/0/app.gosl.mirkfall/app_flutter/mirkfall.db poc-<device>.db`
   - **iOS:** Xcode -> Window -> Devices and Simulators -> select device ->
     MirkFall -> Download Container -> unzip, locate `Documents/mirkfall.db`.
10. Run the plotter:
    ```
    python tool/plot_session_fixes.py --db poc-<device>.db --session-id <sess_...>
    ```
    The script prints the stats that populate the entry below and writes
    a PNG under `docs/poc-artifacts/`.
11. Copy the PNG into `docs/poc-artifacts/` if the plotter wrote it
    elsewhere. Commit PNG + the filled-in entry in the same change.

## Acceptance criteria (per entry)

- [ ] At least **50 fixes** recorded during the 30-minute window.
- [ ] Max interval between consecutive fixes **< 3 minutes** (no silent
      background kill).
- [ ] Last fix timestamp > (start + 29 minutes) — session survived the
      whole walk.
- [ ] Plot visually coherent — the trajectory matches the walked route,
      with no aberrant jumps far from the real path.
- [ ] Persistent notification visible during the walk; dismissed on Stop.

If any criterion fails, mark verdict `FAIL` and document the observed
behaviour in the Notes field. A FAIL does not close Plan 05-06 — it
surfaces to the Phase 06 Review Gate as a finding requiring fix.

## OEM coverage note

Per `.planning/phases/05-gps-session-lifecycle/05-CONTEXT.md`, the Xiaomi
/ Samsung / Huawei / OnePlus OEM-specific POC runs are **deferred to
Phase 15** (release polish). Phase 05 closes with Pixel-only Android
evidence + iPhone evidence; the ROADMAP Success Criterion #1 is marked
"partial — Pixel validated, OEM-specific verification deferred to
Phase 15".

The manual mitigation path for aggressive OEMs is already shipped:
`OemDetector` + `OemGuidanceScreen` (Plan 05-04) surface `dontkillmyapp.com`
links to the user for Xiaomi / Samsung / Huawei / OnePlus / Oppo. Phase 15
will re-validate on a Xiaomi or Samsung device if the maintainer's device
pool allows.
